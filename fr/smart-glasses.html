Pour les responsables techniques OEM de lunettes connectées

Le runtime spatial que les OEM de lunettes connectées licencient—plutôt que de construire.

RakuAI est un runtime spatial neutre vis-à-vis des fournisseurs de LLM que les OEM licencient plutôt que de construire. Phase 1 : tête de pont Meta Ray-Ban Display PWA — une application web installable depuis le navigateur des lunettes, une surface MCP que Claude, ChatGPT, Gemini ou Copilot peuvent piloter, et un budget thermique/latence calibré pour les affichages portés sur la tête. Feuille de route Phase 2 : Android XR + SDK natif (T3 2026–2027). Nous ne concurrençons pas votre matériel — nous faisons en sorte que les expériences d'IA qui s'y exécutent soient livrées plus tôt.

Rédigé pour les responsables ingénierie et partenariats chez Meta, Samsung, Google, Xreal, Rokid et d'autres acteurs livrant des lunettes en 2026–2027. Honnête sur ce qui est en production, ce qui n'est que référence et ce qui dépend du matériel. Aucun partenariat OEM signé n'est affirmé.

Trois phases. Un seul modèle de licence.

La stratégie go-to-market est échelonnée selon la maturité de la plateforme — pas selon le calendrier de RakuAI. Nous rencontrons votre appareil sur les surfaces publiques qu'il expose déjà, puis approfondissons l'intégration dès que vous êtes prêt.

Phase 1 — Tête de pont

PWA Meta Ray-Ban Display · cible T3 2026

  • Entrée : PWA installable depuis le navigateur intégré des lunettes — sans contrainte de boutique d'applications, sans installation de SDK sur votre appareil
  • Surface MCP : 17 outils natifs + 6 outils passthrough universels, en ligne pour Claude, ChatGPT, Gemini, Copilot
  • Capture : téléphone compagnon comme caméra aujourd'hui ; capture directe depuis les lunettes à l'arrivée de l'API de frames de l'OEM (dépend du matériel)
  • Architecture : téléphone comme ordinateur (le téléphone gère l'IA/physique, les lunettes affichent + traquent)
  • Étiquette honnête : la voie PWA est prête en référence ; aucun co-marketing ni préinstallation avec Meta n'est convenu

Phase 2 — Plateforme

Android XR + SDK natif · 2027

  • Entrée : la même voie PWA se généralise sur la surface Chrome standardisée d'Android XR
  • SDK natif : raku-runtime sous forme de DLL natives avec une couche de session compatible OpenXR — intégrée au compositeur de l'OEM, au pipeline de suivi de la main ou au pilote d'affichage
  • Objectif docs SDK : docs de niveau partenaire T4 2026 ; chemin d'intégration de 8–12 semaines développeur
  • Partenaires : Samsung Galaxy XR + licenciés Android XR ; Xreal, Rokid, Vision Pro browser avec la même offre échelonnée
  • Étiquette honnête : fenêtre de lancement Android XR sujette au calendrier de la plateforme ; aucun OEM signé

Phase 3 — Déploiement

Xreal / Rokid / Galaxy XR / Vision Pro · 2027+

  • Prise en charge des appareils de longue traîne via la voie navigateur
  • Résilience hors ligne / connexion intermittente pour les lunettes d'entrée de gamme
  • Pipeline répétable d'auto-qualification → pilote PWA → SDK
  • Application de lancement co-commercialisée pour un partenaire stratégique de Niveau 3
  • Introductions chaleureuses NVIDIA Inception dans l'écosystème BD OEM XR

Réseau NVIDIA Inception : RakuAI a été accepté dans NVIDIA Inception (mai 2026). Ce réseau fournit des introductions chaleureuses dans l'écosystème matériel XR — de nombreux OEM de lunettes et leurs fournisseurs de silicium/optique sont membres d'Inception ou partenaires NVIDIA.

Intégration OEM, étape par étape.

Faites défiler le chemin de la première conversation à une expérience de lancement co-développée. Sans lock-in de SDK — nous rejoignons votre appareil sur les surfaces web publiques qu'il expose déjà.

1

Se connecter via la PWA

RakuAI s'installe comme PWA depuis la surface web de votre appareil — sans contrainte de boutique d'applications. Vos lunettes chargent rakuai.com/capture-app/ et le runtime est actif.

  • Entrée application web installable, aujourd'hui
  • Voie client WebXR / WebGPU
  • Aucun shell natif requis pour démarrer
2

Piloter avec n'importe quel LLM via MCP

Claude, ChatGPT, Gemini ou Copilot pilotent le runtime via les 17 outils MCP natifs — via un pont stdio vers l'appareil compagnon ou en HTTPS vers le MCP hébergé.

  • 17 outils MCP natifs (5 lecture seule + 12 mutation)
  • 6 outils adaptateurs xr_ pour les flux lunettes
  • Neutre vis-à-vis des fournisseurs — l'utilisateur choisit le LLM
3

Capturer l'espace de l'utilisateur

Les images + IMU transitent vers la reconstruction cloud et reviennent en quelques minutes sous forme de splat gaussien explorable — depuis le téléphone compagnon aujourd'hui, directement depuis les lunettes quand votre API caméra arrive (dépend du matériel : nécessite l'API de source de frames de l'OEM).

  • Reconstruction GLOMAP + Brush, sortie .spz
  • Capture téléphone compagnon disponible maintenant
  • Directement depuis les lunettes sur la voie T3 2026
4

Respecter l'enveloppe thermique

Le téléphone comme ordinateur maintient les lunettes au frais : le téléphone gère physique et IA, les lunettes rendent et traquent, le lien transporte des deltas de scène — pas de vidéo. Trois modes thermiques s'adaptent à l'appareil.

  • Objectif 18 ms mouvement-photon
  • ~20 Hz synchronisation delta de scène, 400–1400 o
  • Modes Performance / Équilibré / Faible consommation
5

Co-livrer sous NDA

Accès NDA mutuel aux outils adaptateurs xr_, à l'aperçu capture-depuis-lunettes et au pipeline de splat WebGPU — avec un chemin vers une expérience de lancement co-commercialisée pour votre appareil.

  • Aperçu SDK OEM sous NDA
  • Licence de brevets au niveau architecture disponible
  • NDA mutuel en jours, pas en semaines

MCP depuis les lunettes, en un seul schéma.

Un LLM sur les lunettes communique avec la surface MCP de RakuAI via la PWA — soit par un pont stdio vers l'appareil compagnon, soit via HTTPS vers le MCP hébergé. Le runtime répond en primitives spatiales : ancres, poses, requêtes de scène, références de splat.

Les deux moitiés sont réelles et livrées. Le serveur MCP de production expose 17 outils natifs (5 en lecture seule + 12 de mutation) — voir spatial-engine.html pour l'inventaire complet. Les ~12 outils pertinents pour les lunettes se répartissent ainsi : 6 outils principaux de capture/scène (documentés sur la page MCP) et 6 outils adaptateurs xr_ (session, pose de tête, entrée et primitives d'ancrage pour les flux pilotés par les lunettes). Le MCP hébergé s'exécute sur Azure Container Apps en East US.

Trois colonnes, trois horizons. Nous ne promettons pas ce que nous ne pouvons pas livrer, et nous signalons ce qui dépend d'un comportement de plateforme que nous ne contrôlons pas.

  • Entraînement de splat côté client en WebGPU (voie sans aller-retour cloud)
  • Requête de scène en direct pendant la capture (pas seulement après capture)

2027

Sous réserve des surfaces de plateforme

  • Enveloppe native visionOS (repli WebXR jusque-là)

  • Application de lancement co-marquée pour un ou plusieurs partenaires OEM
  • Sim-to-real en temps réel pour la robotique + les flux agentiques sur lunettes

Ce que nous ne promettrons pas : qu'un OEM en particulier préinstallera notre PWA, qu'un fournisseur de LLM en particulier passera par nous par défaut, ou que des API caméra que nous ne contrôlons pas existeront selon le calendrier que nous souhaitons. Nous concevons pour les surfaces publiques et laissons les plateformes décider.

Le portefeuille de propriété intellectuelle derrière le runtime.

Cinq brevets US accordés, priorité juillet 2012, portefeuille courant jusqu'en juillet 2032. L'architecture décrite sur cette page est celle que ces brevets avaient anticipée.

  • US9781244B2 Lunettes connectées (accordé oct. 2017)
  • US9210358B2 Console de divertissement (accordé déc. 2015)
  • US9654613B2 Jeu en arène (accordé mai 2017)
  • US10027361B2 Point de service (accordé juil. 2018)
  • US10432772B2 Lunettes (21 revendications)

Ce que couvrent les revendications

  • Communication sans fil bimode (faible latence + haute bande passante)
  • Architecture lunettes connectées + hôte de calcul/téléphone
  • Déport du rendu de l'hôte de calcul vers les lunettes
  • Synchronisation de l'état du monde en multijoueur
  • Streaming de monde (livraison de deltas de scène, pas de vidéo)

Ces brevets sont antérieurs aux Meta Ray-Bans (2021), au Xreal Air (2022), au Meta Ray-Ban Display (2024–25) et à Android XR (2026). Pour les OEM qui souhaitent une voie claire sur les revendications au niveau de l'architecture, nous proposons des licences en complément du partenariat.

Architecture téléphone-comme-calcul.

La répartition que les brevets de 2012 avaient anticipée — aujourd'hui la seule architecture qui tient dans l'enveloppe thermique des lunettes.

Le téléphone fait tourner le monde

Physique (60 Hz fixe), comportement IA, état du jeu, streaming d'assets.

Les lunettes affichent le monde

Prédiction de pose locale, rendu fovéal, IMU + odométrie visuelle.

La liaison fait le reste

Delta de scène en Wi-Fi 7 (avec repli optique lorsque disponible), liaison montante des capteurs, aucun streaming vidéo requis.
