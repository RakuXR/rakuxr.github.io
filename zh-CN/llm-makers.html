为什么 MCP 原生对 LLM 厂商至关重要

1. 接入一次，驱动一切。

如果你的助手会说 MCP，它就会说 RakuAI。一份经过版本管理且稳定的 JSON-Schema 契约：17 个原生工具（5 个读取 + 12 个变更），外加从 api.rakuai.com 提供的 ~9,500 个端点中提取的 ~147 个透传端点。无需定制集成，无需按厂商分叉。

2. 设计上多厂商。

该契约对 Claude、ChatGPT、Gemini 和 Copilot 完全相同 — 由架构保证一致。没有独家协议。各厂商的端到端一致性测试正在进行中；在对任何厂商声称已验证对等之前，我们会先公布结果。

3. 仅推理。我们不训练。

RakuAI 是你的模型在推理时调用的空间运行时。我们不持有任何模型权重，不针对你的数据运行训练任务，也不微调任何东西。你的数据始终属于你。这是一条不可逾越的红线。

4. 生产环境默认拒绝。

读取工具（get_scene_state、list_objects、get_metrics、get_world_model、get_render_target）已在生产环境上线。变更工具（set_render_target、start_simulation 及另外 10 个）默认沙箱化并记录审计日志。

第 1 阶段 — 拿下一个连接器 + 首家实验室

我们正处于实验室合作战略的第 1 阶段。目标是在某家实验室的工具目录中拿下一流的 MCP 连接器条目，并共同运行一个可复现的端到端演示。今天尚无已签约的实验室合作伙伴 — 这是目标，而非现状。

对比 Unity

Unity 刚刚发布了 Native MCP。它适用于设计时的场景操作：Claude Desktop 构建场景，Unity 进行渲染。RakuAI 的 MCP 运行在仿真时。你的模型不是在生成场景文件 — 而是在逐帧驱动世界。属于不同的范畴，二者都成立。

适配器契约 （路线图）

规划中的扩展面：合作伙伴交付适配器，将模型特定的输出转换为 RakuAI 场景契约。适配器接口正在设计中；目前尚无任何上线的适配器。

  • VeoEnvironmentAdapter — 规划中：Google 的 Veo （路线图）
  • RunwayEnvironmentAdapter — 规划中：Runway （路线图）
  • YourModelAdapter — 你的模型，待契约发布后

适配器契约设计正在进行中。任何能产出帧 + 场景增量的模型都将能够成为适配器。我们会在请求实现之前公开发布规范。

多厂商的实践

我们的开发循环每天并行使用四家 AI 厂商：

  • Claude Code 编写我们运行时的大部分代码
  • Gemini 审查 PR
  • ChatGPT 作为架构的"小黄鸭"陪聊
  • GitHub Copilot 捕捉其他工具遗漏之处

我们不忠于任何单一模型。我们的运行时也是如此。

诚实现状： MCP 契约对每家厂商由架构保证一致。各厂商的端到端一致性测试 — 加载世界、读取状态、操作、验证往返 — 是尚未完成的第 2 阶段交付物。我们会公开发布结果。请勿将"多厂商"视为今日已验证对等的声明。
