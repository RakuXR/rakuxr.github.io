为什么 MCP 原生对 LLM 厂商很重要

1. 你的模型直接就能用。

只要你的助手会说 MCP，它就会说 RakuAI。六个工具。对每个模型都是同一份契约。无需自定义集成。

2. 从设计上即多厂商兼容。

Claude、ChatGPT、Gemini、Copilot——它们驱动 RakuAI 的方式都一样。你的模型比拼的是意图质量。

3. 生产环境默认拒绝。

审计日志记录。Schema 校验。只读工具在生产环境中可用；变更类工具仅在沙箱/开发环境中使用。

架构

能做到什么

  • 先观察再开口的 NPC——你的模型每一帧都读取场景状态
  • 围绕真实 GPS 坐标重新成形的体验
  • 运行在模拟步进上的世界模型——你的模型是大脑，而非内容供应商
  • 跨平台交付——智能眼镜、头显、Web、移动端

对比 Unity

Unity 刚刚发布了 Native MCP。它擅长设计时的场景操作：Claude Desktop 构建场景，Unity 渲染它。而 RakuAI 的 MCP 运行在模拟时刻。你的模型并非生成一个场景文件——而是逐帧驱动世界。不同的品类，二者都成立。

适配器契约

在六个核心工具之外，合作伙伴可交付适配器：

  • VeoEnvironmentAdapter — Google 的 Veo
  • RunwayEnvironmentAdapter — Runway
  • YourModelAdapter — 你的模型

适配器契约是公开的。任何能产生帧 + 场景增量的模型都可以成为一个适配器。

多厂商实证

我们的开发流程并行使用四家 AI 厂商：

  • Claude Code 编写我们运行时的大部分代码
  • Gemini 评审 PR
  • ChatGPT 充当架构上的"小黄鸭"
  • GitHub Copilot 捕捉其它工具遗漏之处

我们不忠于任何单一模型。我们的运行时也一样。
