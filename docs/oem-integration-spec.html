OEM Integration Spec - RakuAI for Smart Glasses

This is the engineering brief for smart-glasses OEM tech leads evaluating RakuAI as a runtime partner. It covers architecture, hardware envelope, three integration tiers, the MCP tool surface your users' LLMs will see, preliminary licensing, and how much engineering effort each path actually takes. Where something hasn't shipped yet, we say so.

Member of Read more ↓

Why integrate with RakuAI

Three reasons we're worth a half-day evaluation, written for a tech lead.

Any LLM your user prefers, spatially aware in ~5 lines

Claude, ChatGPT, Gemini, Copilot - whichever the user has signed into - connects to RakuAI's MCP server and immediately gets tools to query the scene, capture a space, and drive an XR session. You don't pick a model for them.

MCP-first means no model lock-in

The tool surface is Model Context Protocol, the same standard Anthropic, OpenAI, Google, and Cursor are converging on. If you ship glasses on top of RakuAI, you inherit a runtime that any future LLM can drive without you re-integrating.

Phone-as-compute keeps glasses thermals manageable

The heavy work - reconstruction, semantic indexing, LLM tool dispatch - runs on the paired phone (PWA) and on our Azure backend. The glasses do display, IMU, and a thin client. Your TDP budget stays inside the envelope.

Architecture

A 4-node topology: glasses (thin display + IMU), paired phone (PWA + heavy compute), RakuAI MCP relay (Azure), and the user's chosen LLM.

Glasses talk only to the paired phone (Bluetooth for low-bandwidth deltas, USB-C for bulk capture upload). The phone runs the Raku Capture PWA and is the only node that holds a session with the MCP relay - which in turn forwards tool calls to whichever LLM the user has configured. RakuAI never sits between the user and their model.

Hardware envelope

What RakuAI assumes about the devices in the loop - and, just as important, what we deliberately do not require.

ComponentWhat we assume
Glasses 90Hz display, IMU, optional RGB camera, 4-8 hr battery, ~5-15W TDP. Single embedded browser (for Tier 1) or a thin native client (Tier 2). No dedicated AI accelerator required.
Paired phone WebGPU-capable handset - iPhone 14 or newer, Pixel 7 or newer, recent Galaxy. 6+ GB RAM. Dual rear cameras preferred for capture quality. PWA install path.
Connectivity Bluetooth 5.0+ for delta updates and control envelope (low bandwidth). USB-C or fast Wi-Fi to the phone for bulk capture upload. Phone needs network (LTE/5G/Wi-Fi) to reach the MCP relay.
What we don't require No dedicated on-glasses AI chip. No always-on compute. No custom OS - we run on any system that can host an embedded browser (Tier 1) or load native libraries (Tier 2). No proprietary anchor format.

Integration paths - three tiers

Pick the depth that fits your timeline. Most partners start at Tier 1 to validate the experience, then graduate to Tier 2 for shipped product.

Tier 1

Web-app integration

Your glasses' embedded browser opens the Raku Capture PWA at a known URL. The user signs in with whichever LLM account they prefer; the PWA negotiates the MCP session. Zero SDK install on the glasses. This is the Meta Ray-Ban Display-style path.

Status: Q3 2026 target, no pilot shipped yet.
Cost to integrate: 1-2 dev-weeks on your side.
Tier 2

Native SDK

raku-runtime ships as 18 native DLLs (~30,738 C++ exports) with an OpenXR-compatible session layer. For OEMs who want the runtime threaded directly into their compositor, hand-tracking pipeline, or display driver. Includes the MCP server.

Status: Engine is real and CI-green; partner-grade SDK docs land Q4 2026.
Cost to integrate: 8-12 dev-weeks.
Tier 3

Custom partnership

Bespoke pilot for marquee partners. Co-engineering, exclusive features (custom MCP tools, branded capture flow, joint go-to-market). Reserved for OEMs committing to launch-partner status on a flagship product.

Status: Discussions open.
Cost to integrate: 6+ months, jointly staffed.

The MCP tool surface (what the user's LLM gets)

When the user's LLM connects through the MCP relay, it sees a stable set of tools organized into these categories. The authoritative list is the tool schema file checked into the runtime repo.

Capture

Drive a capture session, observe progress, ingest world-model frames, set render targets. The "scan a space" surface.

Scene query

Read-only inspection of the scene graph - what objects are present, where, with what properties. Used by the LLM to answer "what's in the room?"

Spatial reasoning

Higher-level questions: distance between two anchors, occlusion, reachability, "is the user looking at X?" Mutations to the scene graph live here too.

XR session

Headset / glasses session lifecycle, head pose, controllers, anchors. The bridge between the LLM and what the user actually sees through the display.

Telemetry

Frame time, memory, draw calls, tool-call latency. Lets the LLM (or your engineering team) reason about perf without leaving the session.

Universal passthrough

api_call / engine_call expose any of the ~9,500 REST endpoints and ~10,328 engine functions. Power-user escape hatch when a native tool doesn't exist yet.

The authoritative tool list lives in src/mcp/tool_schemas.json in github.com/RakuXR. As of Wave 7Z (May 2026) the production server ships 17 native tools plus 6 universal passthrough tools - see the engine page for the full breakdown. Counts evolve with the runtime; don't pin your integration to a static number, point at the schema file.

Licensing model

Three tiers, with the same caveat upfront: pricing is preliminary until we sign an LOI together.

Terms preliminary - final pricing is negotiated at letter-of-intent stage and varies by tier, volume, and partnership scope.

Pilot

Free during the partner-pilot phase. 90 days, capped at 10,000 monthly active devices. Designed for evaluating fit before either side commits to revenue terms.

Standard

Per-device royalty model once you ship. Specifics negotiated per deal; volume curves and feature tiers are part of the conversation. Typical structure: a base per-device fee plus optional capability add-ons (e.g., GPU reconstruction credits, premium MCP tools).

Strategic

Volume and exclusivity discounts for OEMs committing to launch-partner status on a flagship product. Co-marketing, joint roadmap, and (where it makes sense) features held exclusive to the launch window.

What's not in this spec yet

Honest disclosure before you take this to your VP. None of this is hidden - we just call it out so you don't get surprised by it three weeks into evaluation.

  • No Tier 1 pilot has shipped to a partner yet. Q3 2026 is the target window, with Meta Ray-Ban Display as the natural fit; no signed launch commitment exists at the time of writing.
  • The native SDK (Tier 2) is in alpha. The runtime is real - 18 DLLs, CI-green on Windows / Linux / macOS - but partner-grade documentation, samples, and a stable C ABI surface are being assembled through Q4 2026.
  • Pricing is preliminary. Every number in the licensing section is a starting point; final terms come out of an LOI conversation, not a price sheet.
  • The MCP tool surface evolves. The count and names you see today (17 native + 6 universal) will grow. Integrate against the schema file, not against a memorized list.
  • Real GPU reconstruction is gated on cloud credits. Production splat training currently lives in the Phase F2 program (Azure A10 + AWS G5 + Nebius H100); local CPU SfM and fallback paths ship today.
