為什麼 MCP 原生對 LLM 廠商至關重要

1. 接入一次，驅動一切。

如果你的助理會說 MCP，它就會說 RakuAI。一份經過版本管理且穩定的 JSON-Schema 合約：17 個原生工具（5 個讀取 + 12 個變更），外加從 api.rakuai.com 提供的 ~9,500 個端點中擷取的 ~147 個透傳端點。無需客製整合，無需依廠商分叉。

2. 設計上多廠商。

該合約對 Claude、ChatGPT、Gemini 和 Copilot 完全相同 — 由架構保證一致。沒有獨家協議。各廠商的端對端一致性測試正在進行中；在對任何廠商宣稱已驗證對等之前，我們會先公布結果。

3. 僅推論。我們不訓練。

RakuAI 是你的模型在推論時呼叫的空間執行階段。我們不持有任何模型權重，不針對你的資料執行訓練工作，也不微調任何東西。你的資料始終屬於你。這是一條不可逾越的紅線。

4. 生產環境預設拒絕。

讀取工具（get_scene_state、list_objects、get_metrics、get_world_model、get_render_target）已在生產環境上線。變更工具（set_render_target、start_simulation 及另外 10 個）預設沙箱化並記錄稽核日誌。

第 1 階段 — 拿下一個連接器 + 首家實驗室

我們正處於實驗室合作策略的第 1 階段。目標是在某家實驗室的工具目錄中拿下一流的 MCP 連接器條目，並共同執行一個可重現的端對端示範。今天尚無已簽約的實驗室合作夥伴 — 這是目標，而非現狀。

比較 Unity

Unity 剛剛發布了 Native MCP。它適用於設計時的場景操作：Claude Desktop 建構場景，Unity 進行算繪。RakuAI 的 MCP 執行於模擬時。你的模型不是在產生場景檔 — 而是在逐幀驅動世界。屬於不同的範疇，兩者都成立。

轉接器合約 （路線圖）

規劃中的擴充面：合作夥伴交付轉接器，將模型特定的輸出轉換為 RakuAI 場景合約。轉接器介面正在設計中；目前尚無任何上線的轉接器。

  • VeoEnvironmentAdapter — 規劃中：Google 的 Veo （路線圖）
  • RunwayEnvironmentAdapter — 規劃中：Runway （路線圖）
  • YourModelAdapter — 你的模型，待合約發布後

轉接器合約設計正在進行中。任何能產出幀 + 場景增量的模型都將能夠成為轉接器。我們會在請求實作之前公開發布規格。

多廠商的實踐

我們的開發循環每天並行使用四家 AI 廠商：

  • Claude Code 撰寫我們執行階段的大部分程式碼
  • Gemini 審查 PR
  • ChatGPT 作為架構的「小黃鴨」陪聊
  • GitHub Copilot 捕捉其他工具遺漏之處

我們不忠於任何單一模型。我們的執行階段也是如此。

誠實現狀： MCP 合約對每家廠商由架構保證一致。各廠商的端對端一致性測試 — 載入世界、讀取狀態、操作、驗證往返 — 是尚未完成的第 2 階段交付物。我們會公開發布結果。請勿將「多廠商」視為今日已驗證對等的宣稱。
