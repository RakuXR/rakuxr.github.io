Bring Your AI. We Bring the Runtime.

May 17, 2026·Kevin Griffin

The LinkedIn page used to say “describe any game, play it instantly.” I’m changing it this weekend. Not because it’s a bad tagline — it tested fine — but because it overpromises in a way that hurts us the moment anyone tries to use it.

“Any game” needs infinite assets. Models. Audio. Behaviour scripts. Art direction. Stories. None of that comes from the engine. We ship 10 templates with assets, and inside those templates a generative model can do a lot — change difficulty curves, swap mechanics, retune NPC personalities, rewrite quest text, retarget the music. That’s the honest envelope. Outside the templates, the user has to bring something. We don’t pretend otherwise anymore.

The replacement framing is the one that’s been quietly true since March, and as of this Saturday is end-to-end provable in production:

Bring your AI. We bring the runtime.

The AI you already trust — ChatGPT, Claude, Gemini, an in-house lab model, whatever speaks MCP — is the thing that designs and customises the game. RakuAI is the deterministic-authority runtime underneath. Physics, scoring, audio, XR, multiplayer state. The parts that have to be correct. The parts the AI shouldn’t be guessing at.

Why the framing change matters now

A week ago I wrote about the six MCP tools and what the adapters unlock. The post said the next move was one real adapter end-to-end, plus three pieces of hardening. All four landed in the seven days since.

RunwayEnvironmentAdapter shipped with a real HTTP client, retry, audit log, and a runnable demo script. Veo, Marble, and Genie adapters followed within forty-eight hours. The production deployment harness shipped (fly.mcp.toml, Dockerfile.mcp, service.py with /healthz, /readyz, /v1/tools, /v1/execute_tool, graceful drain on SIGTERM). Multi-provider fallback routing landed — when a primary adapter fails, the runtime auto-dispatches to a configured fallback. The adapter contract is published in raku-sdk/docs/adapters/CONTRACT.md, with runway.md and veo.md as reference implementations.

This Saturday morning I drove the whole chain end-to-end against raku-api.fly.dev and raku-mcp.fly.dev from a fresh shell. Register an account, accept the NDA, get admin-approved, mint an API key, exchange the key for a JWT, hit /mcp/ with initialize, then tools/list, then tools/call get_metrics, then tools/call get_scene_state, then tools/call start_simulation — which correctly came back denied because production-mode mutation gating works. The 6 tools the previous post promised are there, deny-by-default is enforced at the wire, and any agent that speaks MCP can drive the runtime today without a custom integration.

That changes what “AI-native game engine” means for us. It’s no longer aspirational. The runtime is the thing partners actually drive.

What “your AI” can do

Within a template — pick one, say survival — your AI can retune difficulty curves frame by frame using the DDA hooks (Dynamic Difficulty Adjustment is a runtime primitive, not a feature). Adjust NPC personalities; the engine ships the personality model, your AI parameterises it. Rewrite quest text and dialogue trees inline. Swap the music adaptive layer. Spawn new mechanics from the API surface — there are 2,718 C functions across 16 subsystems and pretty much every gameplay knob is exposed. For visual-generative use cases, push frames through ingest_frame from Runway or Veo or your own model. The engine handles scene composition; the model handles imagination.

What it cannot do, today: invent a 3D character model from nothing. Compose original orchestral music from nothing. Write a 20-hour open-world narrative from nothing. The model labs are getting better at all of that, but the honest version is the engine is provider-agnostic about who solves those problems. We hand off cleanly when the model’s good enough; we don’t pretend we are the model.

What this means for partners

If you’re a model lab and your model produces visuals, audio, behaviour graphs, or any structured world-model output: you can write an MCP client this weekend, point it at https://raku-mcp.fly.dev/mcp/, and have your model driving a real game runtime within an afternoon. The adapter contract is published. The four lifecycle methods are documented. The two reference implementations (Runway, Veo) are open source. There’s no partner application form, no NDA gate on the MCP side. Build the adapter, send a PR.

If you’re a developer who wants to use RakuAI as the runtime for an AI-driven game: same answer with one extra step. Get a raku_ API key via rakuai.com/developers, exchange it for a JWT, point your MCP client (Claude Desktop, Cursor, your own agent) at https://raku-mcp.fly.dev/mcp/, and the 6 tools appear as first-class capabilities your AI can call. We have a step-by-step at developers/claude-desktop.html.

If you’re a studio: the runtime is free, the API is open, the .raku file format is documented, and the patents covering AI-driven game personalisation, companion gaming, and adaptive difficulty are licensed under the same open terms. Bring your AI. We bring the runtime. The handoff is the product.

What I’m not going to say

I’m not going to say “describe any game, get any game in 60 seconds.” Not because it’s never true, but because it’s not the contract. The contract is that the AI is the designer, the engine is the deterministic substrate, and the boundary between them is six tools. That’s the thing we can guarantee. The thing we can deploy. The thing we can hand to a partner with a straight face.

The Saturday-in-motion habit is checking whether the work product matches the marketing copy. This weekend, the marketing copy moved.

Bring your AI.

We bring the runtime.
